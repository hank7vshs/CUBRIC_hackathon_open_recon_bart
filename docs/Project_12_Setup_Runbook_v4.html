Create the GitHub repo

03

Initialize the local project

  1. Create the project folder

    mkdir -p ~/dev/CUBRIC_hackathon_open_recon_bart
cd ~/dev/CUBRIC_hackathon_open_recon_bart

    (Copy the project's files into this folder before continuing — the original base repo content.)

  2. Initialize git and make the first commit

    git init
git add .
git commit -m "Initial commit"
git branch -M main

  3. Point it at the GitHub repo (HTTPS for now — swapped to SSH in §5)

    git remote add origin https://github.com/hank7vshs/CUBRIC_hackathon_open_recon_bart.git
04

SSH key setup & GitHub connection

  1. Check for an existing key first

    ls ~/.ssh

    Look for id_ed25519 / id_ed25519.pub. If already present, skip straight to step 3 (adding it to the agent).

  2. Generate a new key, if needed

    ssh-keygen -t ed25519 -C "hank7v@gmail.com"

    Accept the default location (~/.ssh/id_ed25519) when prompted.

  3. Start the SSH agent and add the key

    eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
ssh-add -l

    ssh-add -l confirms the key is actually loaded into the agent — an empty result means it didn't take.

  4. Copy the public key

    cat ~/.ssh/id_ed25519.pub

    Copy the full output. In GitHub: Settings → SSH and GPG keys → New SSH key → paste it in, give it a recognisable title (e.g. "CUBRIC hackathon VM").

  5. Verify the connection

    ssh -T git@github.com

    Expected response: a message confirming successful authentication as your GitHub username — not a shell prompt (GitHub deliberately doesn't grant shell access over this connection).

05

Switch the remote to SSH & push

  1. Confirm the current remote

    git remote -v

    Should still show the HTTPS URL from §3.

  2. Swap it to SSH

    git remote set-url origin git@github.com:hank7vshs/CUBRIC_hackathon_open_recon_bart.git

  3. Push

    git push -u origin main

    -u sets main to track origin/main, so a plain git push is enough from here on.

Per the earlier decision: no second VS Code window. A plain terminal (not VS Code's integrated one) runs the build/test loop, leaving the native VS Code + Claude Code window untouched throughout.

cd ~/dev/CUBRIC_hackathon_open_recon_bart

Inside the running container, verify whatever's relevant to the current change, e.g.:

If it works, the same Dockerfile will work identically for every student, regardless of their host OS (Framework Description doc, §3b). If it doesn't, edit the Dockerfile in the native VS Code window and repeat this section.

Install script & from-scratch replay — original plansuperseded, see §12/§13

Current state: working inside the student-target devcontainer, missing several tools students will need. Plan, in order:

  1. Install the missing tools manually, live, in the current container

    Whatever's needed beyond what's already there — MRtrix3, conda, and anything else identified as required. This is exploratory: the goal right now is confirming what works and which exact commands, not durability. Expected and fine for this to be lost later (see §10's fixnote).

  2. Turn the working commands into a script

    Once each tool installs cleanly by hand, fold the exact commands into install_extra.sh, following the same shape already used for BART — non-interactive flags throughout (DEBIAN_FRONTEND=noninteractive, -y, conda's -b/-y, no bare interactive installer scripts). Add matching entries to /etc/environment, /etc/profile.d/, ~/.bashrc, and devcontainer.json's remoteEnv for any new PATH additions, per the existing BART pattern.

  3. Restart the container — deliberately discarding the manual installs

    docker stop <container-id>
docker rm <container-id>

    This is the actual test. Nothing manual survives; the only thing left to reproduce the environment is the script itself.

  4. Reopen in Container — let install_extra.sh run from scratch

    Ctrl+Shift+P → Dev Containers: Reopen in Container. postCreateCommand re-runs the updated script automatically on this fresh container.

  5. Verify the replay matches the manual result

    Re-check every tool added in step 1 — version, basic functionality — inside this freshly-built container. Any mismatch means the script is missing a step that was done by hand and forgotten; fix the script, repeat from step 3.

  6. Once verified: fold into the main documents

    With the script confirmed reproducible, this becomes the authoritative record — merge the relevant parts into the Framework Description doc (teaching/logic) and this Setup Runbook (operative steps), rather than leaving them as a separate working log. Candidate for extending into the actual hackathon-day sequence once this environment-setup portion is solid.

outcome v4 MRtrix3 was tested exactly per this plan (installed manually, worked cleanly). But the "fold into install_extra.sh" step in item 2 was deliberately not taken for MRtrix3 or Copilot — see the revised callout in §10 for why, and §13 for the actual student-facing flow that replaced it. Running test_server.sh / test_docker.sh end-to-end for the first time since the Python bump surfaced four distinct issues. None turned out to be Python-3.11-related — worth recording precisely, since three of the four will resurface for anyone testing via plain docker build/docker run instead of VS Code's devcontainer lifecycle.

SymptomRoot causeFix
PermissionError opening test data, then again creating out.h5 in client/ COPY . /container in .devcontainer/Dockerfile defaulted to root:root ownership — COPY ignores the active USER directive unless --chown is given explicitly. recoon could read (world-readable) but not write. ImportError: The 'bart' Python module could not be found BART is deliberately not baked into .devcontainer/Dockerfile — it's built by install_extra.sh via postCreateCommand, which only fires through VS Code's "Reopen in Container" lifecycle. A container started via plain docker build/docker run never triggers it. bash /container/.devcontainer/install_extra.sh manually. Again, only relevant to manual-CLI testing — automatic for students via the normal flow.
dockerd fails: failed to mount overlay: operation not permitted / iptables Permission denied The nested Docker-in-Docker daemon needs elevated capabilities (CAP_NET_ADMIN for iptables/NAT, overlay filesystem mounts) that an unprivileged container doesn't grant even to its own root user. devcontainer.json's runArgs: ["--privileged"] supplies this automatically; a plain docker run without that flag does not. Net takeaway: only the first row (file permissions) was an actual repo bug, now fixed at the source. The other three are artifacts of testing outside VS Code's lifecycle for diagnostic speed (§8's "no second VS Code window" approach) — students using the standard "Reopen in Container" flow won't hit rows 2–4 at all.

Student walkthrough — first test runv4

What a student actually does, once they've opened this repository in VS Code and selected Reopen in Container — the normal, supported flow (unlike §8-§12 above, which deliberately used plain CLI for fast supervisor-side iteration). Because of that, none of §12's rows 2–4 apply here: postCreateCommand runs automatically, so BART is already built and containerEnv is already set by the time the first terminal opens.

Use three terminals, not one. run_server.sh blocks in the foreground waiting for a single client connection — backgrounding it with & works, but swallows its live output, which is exactly where the real error shows up when something goes wrong (see §12). Open three terminal panes in VS Code (Terminal → New Terminal, or the split-terminal icon) — one for ad-hoc commands, one dedicated to the server, one dedicated to the client.

  1. Terminal 1 — manual installs (once per container lifetime)

    GitHub Copilot CLI and MRtrix3 are deliberately not pre-installed (§10) — this is the hands-on exercise. Work through .devcontainer/manual_install_reference.sh command-by-command rather than running it as a script; it's a reference, not something to bash wholesale.

  2. Terminal 2 — server, Terminal 3 — client: start with i2i_invertcontrast

    No BART dependency at all — the simplest possible sanity check that the devcontainer, the MRD client/server protocol, and the module dispatch all work before touching anything BART-related.

    # Terminal 2
cd /container/server
./run_server.sh i2i_invertcontrast

# Terminal 3, once Terminal 2 shows "Serving..."
cd /container/client
./run_client.sh i2i_invertcontrast

  3. Then repeat with r2ci_bart

    Exercises the full stack, including BART's Python bindings via BART_TOOLBOX_PATH (set automatically by run_server.sh itself — no manual export needed here, unlike the MRtrix3 PATH case in §11).

    # Terminal 2
cd /container/server
./run_server.sh r2ci_bart

# Terminal 3
cd /container/client
./run_client.sh r2ci_bart

  4. Once both pass individually: run the full suite

    cd /container/test
./test_server.sh --speed fast

    This is the same check from the main AGENTS.md verification checklist — five fast tests (three r2ci_bart, two i2i_invertcontrast), each printing an NRMSE value under its error margin followed by Test passed successfully!. Confirmed end-to-end on Python 3.11 v4:

Supervised_RNS_dMRI dependency check (supervisor-only)v4

Not for students. Same treatment as MRtrix3 (§11/§13): installing this manually, in the devcontainer, is the hackathon exercise — figuring out its dependency chain and integrating it into a real OpenRecon module is the actual challenge, not something to hand over as a script. This section exists purely so a rebuilt/wiped devcontainer can be brought back to this checkpoint quickly, without re-deriving the same commands from scratch.

This installs the package and its full dependency chain (DiffSimGen, the dmipy fork, dipy, snakemake==7.25.0, snakebids==0.14.0, scikit-learn, scikit-image, nibabel, scipy, PuLP, docutils — all resolved transitively from its own pyproject.toml). It does not clone the source — pip install git+... builds from a temporary internal clone and discards it, leaving only the installed package, importable and runnable via its console-script entry point, with no browsable/editable source folder. A separate git clone (of both this repo and DiffSimGen) is a distinct, later step for whoever actually adapts the code into server/modules/supervised_rns_dmri/.

# Compiler toolchain — needed for a few packages in the chain (datrie,
# connection-pool, stopit, docopt, some cvxpy solver backends) that build
# native extensions from source rather than shipping prebuilt wheels.
# Likely a no-op in this devcontainer already (gcc/g++/make are already
# installed), but harmless and confirmed needed from a bare base image.
sudo apt-get update -qq
sudo apt-get install -y --no-install-recommends build-essential

# Installs Supervised_RNS_dMRI + its full dependency chain in one shot
pip install --no-cache-dir git+https://github.com/Bradley-Karat/Supervised_RNS_dMRI.git

Confirmed working in throwaway python:3.11-slim-bullseye friction tests against the pinned numpy==2.4.2 and matplotlib==3.9.2 (§10) before running it here for real — dmipy specifically hard-requires Python <3.12 in its own package metadata, which is why the Python bump (§10) had to happen before any of this could even install.

Note — pip install git+... vs. git clone, and why not Poetry:

pip install git+https://... clones the repo into a temporary directory, builds a wheel from its pyproject.toml, installs that wheel into site-packages, then discards the clone — you get an importable/runnable package, but no browsable or editable source folder. That only works at all because both Supervised_RNS_dMRI and DiffSimGen declare proper packaging metadata ([tool.poetry] + poetry-core); a repo without that (loose scripts/notebooks, no pyproject.toml/setup.py) can only be used via an actual git clone, since pip would have no way to build a wheel from it. The maintainers' own README only documents git clone + poetry install — partly because their poetry.lock (exact transitive-dependency pinning) only applies when the repo is actually cloned alongside it, and partly because this is research code meant to be read/modified, not consumed as an opaque library — which is exactly what the hackathon exercise requires too.

We deliberately used plain pip install here rather than Poetry, for a reason specific to this codebase's architecture: server.py dispatches every module from one shared Python environment (no per-module venv), so the real question was whether this dependency chain installs cleanly inside that same shared environment alongside the already-pinned numpy/matplotlib. Installing via Poetry would have resolved everything inside its own isolated venv instead, hiding exactly the kind of conflict (§10's matplotlib downgrade) this check exists to surface. Poetry's environment-isolation model also doesn't fit here for the same reason Miniconda was dropped for MRtrix3 (§11) — an extra isolated environment just adds a bridging problem (subprocess calls, poetry run) that this single-process server architecture doesn't need.

15

Module integration roadmap — next steps for studentsv4

Once §13's framework checkpoint passes and §14 confirms the dependency chain installs cleanly, this is the actual hackathon exercise: integrating Supervised_RNS_dMRI's model-fitting logic into a real server/modules/supervised_rns_dmri/ module. This is a roadmap of the logical steps, not a tested recipe — unlike earlier sections, none of this has been built yet.

  1. 1. This is an image-to-image workflow, not raw k-space

    Diffusion model fitting operates on already-reconstructed DWI volumes, not raw acquisitions. i2i_invertcontrast is the template to start from, not r2ci_bart — the module will receive a stream of ismrmrd.Image messages, not ismrmrd.Acquisition.

  2. 2. Bridge the core format mismatch: MRD streams per-volume; the tool expects one 4D file on disk

    MRD sends one ismrmrd.Image message per diffusion volume/slice. Supervised_RNS_dMRI is a BIDS app expecting a single 4D NIfTI (x, y, z, gradient-direction) plus .bval/.bvec sidecar files on disk. Buffer every incoming image for the series (same accumulation pattern i2i_invertcontrast.py already uses), stack once complete, write out via nibabel — already in the dependency chain, no new tool needed for this step.

  3. 3. Where do b-values/b-vectors actually come from, and what do they get used for?

    Format and usage, traced through the actual source (prepare_data_and_noisemaps.py, run_model_fitting.py, normalize_noisemap.py): both files are loaded with plain numpy.loadtxt() — standard FSL-style text, exactly what MRtrix3's mrconvert -export_grad_fsl produces. bvals identifies b=0 volumes (for signal normalization) and groups directions into b-shells; bvecs is only used to direction-average each shell via a spherical-harmonics fit before model fitting — the final trained Random Forest/MLP itself only consumes the normalized, direction-averaged signal, not raw per-voxel gradient directions. The noisemap input is expected as an already-computed MP-PCA estimate (the script loads it, doesn't compute it) — strong evidence MRtrix3's dwidenoise is the intended upstream source.

    Where this metadata comes from on a real scanner: MRD has no dedicated b-value/gradient-direction field — the DICOM instinct is the right one here. It would have to ride in through the same kind of vendor/private-tag mechanism MRD offers: userParameterLong/userParameterDouble/userParameterString at the header level, or per-image Meta attributes (already used elsewhere in this repo — r2ci_bart.py reads BitsStored exactly this way).

    Scope decision for this hackathon: since the deployment target is a server tested against a local .h5 client (§13's own test harness), not a live scanner, solving the genuine custom-field extraction problem is explicitly out of scope here. Simpler options, roughly in order of protocol-realism vs. ease: (a) bundle .bval/.bvec/mask/noisemap files directly alongside the test .h5 in test/data/, read by convention; (b) pass them through the existing ui_data JSON channel, the same mechanism already used for numCoilIntId/picsFlagsId; (c) skip supplying mask/noisemap externally at all — compute them on the fly from the incoming DWI images via MRtrix3 (dwi2mask, dwidenoise). Trade-off to keep visible, not gloss over: option (a) specifically means the resulting module isn't scanner-deployable as-is — a real scanner has no mechanism to drop extra files onto the server's disk at a fixed path. Fine for proving out the integration in three days; worth a one-line note wherever this module's own Readme ends up describing its limitations.

  4. 4. Brain mask and noise map — MRtrix3's actual role

    dwi2mask for the brain mask, dwidenoise's MP-PCA residual for the noise estimate — both feed directly into what prepare_data_and_noisemaps.py already expects as inputs, per point 3 above.

  5. 5. Call the model-fitting logic — vendored function call, not subprocess

    Once the temp BIDS-like folder exists, either call the extracted/adapted Python functions directly (clean, no per-connection subprocess/file-I/O overhead) or shell out to the CLI as a stopgap (subprocess.run([...])) to get something working end-to-end before optimizing.

  6. 6. Convert output parameter maps back to MRD Images

    Supervised_RNS_dMRI writes NIfTI parameter maps to disk — read them back with nibabel, convert to ismrmrd.Image.from_array(...) (mirroring exactly what r2ci_bart.py/i2i_invertcontrast.py already do: setHead(), field_of_view, Meta attributes, image_index), and connection.send_image().

  7. 7. Define scanner UI parameters in doc/appl_spec.json

    Which model to fit, output map selection, etc. — following the existing module pattern.

  8. 8. Test incrementally, cheapest case first

    Get the buffering/format-conversion pipeline working with a pass-through (echo the DWI images back unchanged) before wiring in the actual model-fitting call — isolates "is the MRD↔NIfTI bridge correct" from "does the science work," same philosophy as starting with i2i_invertcontrast before r2ci_bart in §13.
