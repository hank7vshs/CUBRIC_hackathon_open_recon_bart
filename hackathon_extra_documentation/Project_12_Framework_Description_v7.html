The four tiers, at a glance

Everything in this project happens on one of four distinct machines or artifacts. Mixing them up in conversation is the main source of confusion — so this version is organised strictly tier by tier, and each tier answers the same two questions: what's installed there, and whose git identity acts there.

1

Host VM

Ubuntu machine each student starts from. Nothing project-specific yet.

2

Supervisor's devcontainer

Where you amend the recipe before anyone else sees it.

3

Student's devcontainer

Their own instance, built from your pushed recipe.

4

Final image

What students build and hand to end users — never a devcontainer.

Only one thing ever crosses a tier boundary by git: the repo. Nothing else — not a container, not an image, not a login — travels that way.
02

Tier 1 — Host VM

Installed directly on Ubuntu, no Docker involved. This is the only tier that exists before any repo has been cloned — and, for the supervisor specifically, it's also where the repo actually gets edited (see §3a).

Installed packagesGit account in use
Docker Engine · VS Code + Dev Containers extension · git (the binary itself — needed here because the very first git clone happens before a devcontainer exists to clone it from, and because the supervisor edits and commits from here directly, see §3a) Student: their own personal GitHub account, authenticated once (SSH key or HTTPS token) — this is what performs the initial git clone.
Supervisor: whichever account is currently configured natively on this VM — see the account-swap note in §3a if that's currently a restricted company account.
correction vs v3 The v3 table implied git only lived in the devcontainer tier. That's wrong: git has to be present on the host VM too, specifically to perform the first clone that brings the repo (and therefore .devcontainer/) onto the machine in the first place.

Claude Code and Copilot are deliberately not installed here — there is nothing for an editor assistant to do against a host with no repo open yet. They only ever appear at tier 2/3.

03

Tier 2 — Supervisor's devcontainer

Built from .devcontainer/Dockerfile the first time you open the (as-yet ungeneralized) repo and select Reopen in Container. This is a verification tier — where you confirm the recipe actually produces a working environment — not where the recipe gets authored. See §3a for why editing happens one tier down, on the host.

Installed packagesGit account in use
Base image + install_extra.sh together provide: git, Claude Code and/or Copilot, BART (source-built via postCreateCommand), MRtrix (dev use, same mechanism), CUDA toolchain — plus anything you newly add this round (MRtrix build deps, Python packages for the noise-estimator pipeline). See §3b/§11 for which layer each tool belongs in. Your git identity — same account as configured on the host VM (§2), since this is a bind mount of the same working copy, not a separate credential set. Bind-mounted (~/.gitconfig, ~/.ssh) or entered live; never baked into the image.

What you actually do at this tier: Reopen/Rebuild in Container after each meaningful edit, then run whatever checks matter — does MRtrix install cleanly, does copilot --version work, does the network resolve. Once confirmed, you're done with this tier until the next edit worth re-checking; there's nothing here worth preserving between sessions.

practice before publishing Do a clean-clone test before pushing what students will use: clone into an empty folder (not reuse your working copy) and Reopen in Container from scratch. Your long-lived devcontainer may carry state — manually installed packages, cached files — that never made it into the actual Dockerfile. Only a fresh clone proves the recipe alone reproduces a working environment.
a trap to avoid If a tool needs to exist for students, it must be in the recipe — either a RUN line in a Dockerfile, or a command in install_extra.sh (run via postCreateCommand) for source-built tools — not a command executed live in this container's terminal. Anything installed by hand here lives only in this one running instance's writable layer: invisible to git, invisible to students, gone the moment the container is deleted. This is exactly what happened installing Copilot CLI live instead of adding it to the recipe.
03a

Editing the repo happens on the host VM, using its native gitnew

Because tiers 1 and 2 are a bind mount of the identical files — not two separate copies — editing .devcontainer/Dockerfile, server/Dockerfile, or any other repo file from a plain editor on the host VM produces exactly the same result on disk as editing it from a terminal inside the devcontainer. Neither location is more "correct" for the file itself.

But given today's troubleshooting (devcontainer networking, DNS, and installer hangs — all tied to the container/Docker layer), the practical recommendation is: do the actual authoring — Dockerfile edits, module code, docs — on the host VM, using git installed there natively. This sidesteps every container-specific fragility encountered so far, since none of it is relevant to editing text files and committing them.

Host VM — author here

Edit any repo file with a plain editor. Run git add / commit / push with git installed natively on the VM. No devcontainer, no Docker network, no risk of the issues hit today.

Devcontainer — verify here only

Reopen in Container periodically to build-test the recipe. Treat it as disposable: delete and rebuild freely, since nothing worth keeping should ever live only in its writable layer.

action needed before publishing The git account currently configured natively on this host VM is the SHS company account. Company accounts commonly carry organisation-level restrictions (private-by-default repos, SSO enforcement, visibility policies) that can block external students from cloning or using "Use this template," even if the repo itself looks public in the UI. Before pushing the repo students will use, swap the VM's native git identity to a personal GitHub account — the one that will actually own and expose the public template repo — so nothing about SHS's org policies limits student access downstream.

To swap: update git config --global user.email / user.name on the host VM, and re-authenticate (new SSH key or HTTPS token tied to the personal account) before the push that goes out to students. This only affects your commits going forward — it has no bearing on the git identity any student later uses inside their own devcontainer (§4), which is always their own account regardless of whose account pushed the template.

03b

Linux isolation — why what works for you works for every studentnew

The host VM's operating system is irrelevant to whether an install command works. Whatever runs inside a devcontainer runs on the Linux image declared in .devcontainer/Dockerfile — not on the host underneath it. This project's actual base image, confirmed directly from the repo:

FROM python:3.12-slim-bullseye
ENV DEBIAN_FRONTEND=noninteractive

python:3.12-slim-bullseye is Debian 11. Every devcontainer built from this Dockerfile — yours on Ubuntu, a student's on Windows, another's on macOS — is the exact same Debian 11 environment underneath. Docker Desktop, WSL2, or native Linux Docker all present an identical container runtime API; none of them leak host-OS behaviour into the container itself.

Practical consequence: if apt-get install -y mrtrix3 works in your devcontainer, it will work — byte-for-byte identically — in every student's devcontainer, regardless of what they're running Docker on. There is no OS-compatibility question to test for; testing on your one Ubuntu VM is testing for all of them.

The Dockerfile already sets DEBIAN_FRONTEND=noninteractive globally, and pins the Docker-in-Docker version deliberately:

ENV DOCKER_API_VERSION=1.43
...
VERSION_STRING=5:24.0.0-1~debian.11~bullseye

That pin isn't arbitrary — 24.0.0 matches the Docker version actually present on the VA60 scanner's MaRS GPU baseline. This is the same isolation principle one layer further: the devcontainer isn't just OS-independent from the host, it's deliberately made to match the scanner's own environment, so a build tested inside it behaves the same way when it eventually runs on real hardware.

Two valid places to add a new tool — not just Dockerfile RUN lines

Earlier guidance in this doc framed every new dependency as a Dockerfile RUN line. This repo's actual convention, visible in install_extra.sh and its Readme.md, adds a second, equally valid mechanism — worth using for MRtrix3, since it fits the same profile as BART:

MechanismBest forExample already in this repo
RUN lines in DockerfileSmall, stable, version-pinned installs — apt packages, pip packages with fixed versionspip install h5py==3.12.1, ismrmrd==1.14.1
postCreateCommand → install_extra.shTools built from source, version-sensitive, or GPU-conditional — exactly MRtrix3's profileBART: cloned, checked out to v0.9.00, built with make -j12, conditional on OR_GPU_SUPPORT

Both are equally non-interactive and equally reproducible across every student's machine — the choice is about maintainability, not about avoiding prompts. Since MRtrix3 is source-built and version-sensitive in the same way BART is, the existing pattern in install_extra.sh is the natural place for it, following exactly the same shape:

if [ ! -d "/container/mrtrix3" ]; then
    git clone https://github.com/MRtrix3/mrtrix3.git /container/mrtrix3
fi
cd /container/mrtrix3
git checkout <pinned-version-tag>
./configure
./build

sudo bash -c 'echo "MRTRIX_PATH=/container/mrtrix3/bin" >> /etc/environment'
sudo bash -c 'echo "export PATH=/container/mrtrix3/bin:\$PATH" > /etc/profile.d/mrtrix.sh'
echo 'export PATH="/container/mrtrix3/bin:$PATH"' >> ~/.bashrc

Matching remoteEnv entry in devcontainer.json is required too — VS Code injects remoteEnv.PATH into every integrated terminal before any shell starts, which is the one mechanism the other three (/etc/environment, profile.d, .bashrc) can't reach on their own for non-interactive contexts like tasks and launch configs.

gotcha, already hit once this project This repo's install_extra.sh runs sudo update-ca-certificates as its very first line, before any network operation — because host SSL certs are mounted into the container by devcontainer.json, but only get processed into a trusted bundle in postStartCommand, which runs after postCreateCommand. Skip this and the first git clone in the script fails with server certificate verification failed — a different root cause from the DNS/network saga earlier in this doc, but the same category of "looks like a network failure, is actually a host/container timing issue."

"It's installed" and "it's logged in as me" are two separate facts; worth saying out loud at the start of the session.
05

Tier 4 — Final image (shared with users)

Built from server/Dockerfile by docker_build.sh / create_archive.sh, from inside the student's devcontainer. This is the only tier of the four that is ever handed to someone outside the development loop — the scanner, or whoever installs the OpenRecon package.

Installed packagesGit account in use
Runtime-only: Python, PyTorch (or chosen inference library), MRtrix3 if confirmed runtime-required (see §9), pretrained weights fetched at build time, src/ and the new module's code. No git, Claude Code, Copilot, or any dev tooling — none of it belongs in an image that ships to a scanner. None. This artifact is a .tar/zip, not a git object — it has no git identity at all. Its provenance is whatever commit the student's devcontainer had checked out at build time.
Why this row matters: it's the tier most likely to accidentally leak dev tooling if server/Dockerfile is edited carelessly (e.g. copying from a devcontainer base image instead of a clean one). Keep the two Dockerfiles strictly separate — this is the two-Dockerfile discipline already established for the base framework, and it matters more here because this project adds real new runtime dependencies.

VS Code extensions & settings — recipe vs. personalnew

The same "recipe vs. personal" split that governs git identity and dev-tool auth (§3, §4) applies to VS Code itself, and it's easy to assume it doesn't — extensions feel like a personal editor preference, but only some of them actually behave that way.

UI extensions

Themes, icon packs, pure-interface tools. Run on the local VS Code process. These do carry over automatically no matter what you connect to — host or any devcontainer — because they never leave your local machine.

Workspace / remote extensions

Python, Docker, GitLens, Copilot, linters, language servers — anything that actually operates on code. These run inside whatever environment is connected, via a separate VS Code Server spun up inside the container. They do not carry over from host to devcontainer, or from your devcontainer to a student's, automatically.

common misassumption Installing an extension while connected to your own devcontainer does not make it appear in a student's devcontainer, even though both were built from the same .devcontainer/Dockerfile — extensions installed by hand are not part of the recipe unless explicitly declared in it.

Observed directly in this project's own devcontainer — VS Code actually distinguishes two cases for extensions that exist on the host, and shows each one differently in the Extensions panel:

Case A — already installed on the host before the container was created

These were reconciled automatically at container-creation time and appear under INSTALLED — fully active, gear icon available, no prompt. This is the one-time inheritance step.

Case B — added to the host after the container was already open

These appear grayed out under LOCAL - INSTALLED with a blue Install in Dev Container button — detected, but not pushed in automatically. Requires a manual click, and only affects this one running instance.

Neither case helps a student. Both behaviors are reconciliation against your own host's local extension list — convenient for you personally when iterating on your own devcontainer, but a student's host starts with none of your extensions installed locally, so there is nothing for their container to inherit or suggest, in either case. The only mechanism that reaches every student identically is still declaring the extension list in .devcontainer/devcontainer.json, below.

The correct mechanism is to declare the extension list inside .devcontainer/devcontainer.json, exactly the same "baked into the recipe" pattern already used for git, Claude Code, and Copilot in tiers 2/3:

"customizations": {
  "vscode": {
    "extensions": [
      "ms-python.python",
      "ms-azuretools.vscode-docker",
      "GitHub.copilot"
    ]
  }
}

Once this is in the pushed recipe, every student's devcontainer installs the identical extension set automatically the first time it builds — no manual step, no drift between teams.

Settings travel the same way. Personal User settings (host or devcontainer) never travel with the repo. Any editor setting you want shared across the team — formatter, tab width, linting rules — belongs in .vscode/settings.json committed to the repo itself, so it rides along with git and applies inside every student's devcontainer too.
TierWhat a student needs to preinstall on VS Code
1 · Host VMVS Code + the Dev Containers extension only — nothing else. No project-specific extensions belong here.
2/3 · DevcontainerNothing manual — the full extension set arrives automatically via devcontainer.json the moment the container builds, identical for supervisor and every student.

Which repo do I actually push, and what happens when a student clones it?

The repo you push is the one on your host VM (§3a) — there is no separate "distribution copy" to prepare. Whether you last touched a given file from a host editor or from inside your devcontainer's terminal (§3), it's the same bind-mounted files on disk; git sees no difference either way. What matters is simply: commit and push whatever's currently on disk.

So yes — a student who clones it and hits Reopen in Container will build their own, independent devcontainer instance from the identical .devcontainer/Dockerfile. That is not a side effect to work around; it is the intended mechanism (§4 of the earlier version, unchanged): one recipe, shared via git, instantiated separately by every student.

What travels

The repo — .devcontainer/Dockerfile, server/Dockerfile, src/, modules/, docs. Text files, via git push / git clone. Nothing binary, nothing built.

What never travels this way

Your running devcontainer, any built image, your git credentials, your Claude Code / Copilot login. These stay on your machine or are re-created independently on each student's.

mechanism On GitHub, enable Settings → General → Template repository on the repo you push. Students then use the "Use this template" button instead of a plain fork/clone-onto-your-remote — each gets their own repo with its own remote from the start, so nobody can accidentally push back into yours, and tutors can still open each team's repo individually to debug via commit history.

Full flow, corrected

  1. Tier 1 (you): on your host VM, git clone the base SHS repo. Add new tools to the right place: install_extra.sh via postCreateCommand for source-built/version-sensitive tools (MRtrix3, following the existing BART pattern), or Dockerfile RUN lines for small pinned installs (§3b, §11). Edit server/Dockerfile only if a dependency is needed at inference runtime (§10). All editing done with a plain editor and git natively installed on the VM (§3a).
  2. Tier 1 (you), account swap: if the VM's native git identity is currently a restricted company account, switch it to the personal account that will own the public template repo — see the pending action in §3a — before the push in the next step.
  3. Tier 2, verification only: Reopen in Container periodically to confirm the edited recipe actually builds and works — MRtrix installs, tools run, network resolves. Treat the container as disposable; delete and rebuild freely.
  4. Tier 1 (you), again: back on the host VM, clean-clone test into an empty folder, then git commit + push to the public template repo.
  5. Tier 1 (student): on their host VM, install Docker/VS Code/git, then either git clone directly or click Use this template on GitHub — authenticated with their own personal account.
  6. Tier 3: student hits Reopen in Container. Their own devcontainer builds from your recipe — git, Claude Code/Copilot, BART, MRtrix all present automatically. They authenticate their own Claude Code / Copilot session. They add the noise-estimator module, vendor the denoiser source, commit under their own git identity.
  7. Tier 4: from inside their devcontainer, the student runs docker_build.sh / create_archive.sh, using the nested Docker engine, producing the final image — no git identity attached to this artifact at all.

Docker layers — the general model (any project)new

Everything above talks about which Dockerfile a tool goes in. This section is the layer underneath that: how Docker actually turns a Dockerfile into something running, and why "installed by hand in a live terminal is lost on delete" is a mechanical fact, not a rule of thumb. This applies to any Docker project, not just this one.

LevelWhat it isMutable?Survives container delete?
Image layers One immutable, cached layer per filesystem-changing Dockerfile instruction (FROM, RUN, COPY), stacked in order. Each stores only its diff from the layer below. Shared across images/containers that use the same layer. No — read-only ✓ yes — they are the image; deleting a container doesn't touch them
The image The complete ordered stack of layers — a static blueprint. Doesn't run anything. This is what docker build produces and create_archive.sh exports as a .tar. No — immutable ✓ yes
Writable layer One extra read-write layer Docker adds on top when a container starts (docker run / Reopen in Container). Uses copy-on-write: modifying a file from a read-only image layer copies it up here first, leaving the original untouched. Yes ✗ no — destroyed with the container on docker rm
Bind mounts Host filesystem paths made visible inside the container (repo folder, ~/.ssh, ~/.gitconfig). Never copied in, never part of any layer. Yes (they're the live host files) n/a — they live on the host, nothing to lose
The one fact that explains this whole document's discipline: only image layers survive a container delete untouched. Everything else is either transient (writable layer) or external (bind mounts). So "if you want students to have it, it must be in the recipe" isn't a style preference — it's the direct consequence of the writable layer being discarded on docker rm.
In this project, that maps to:Which level
Dockerfile RUN/COPY/ENV — Debian base, Python packages, DinD 24.0.0Image layers (survive rebuild)
postCreateCommand → install_extra.sh (BART build, soon MRtrix)Technically the writable layer — but re-run automatically on every fresh container, so it behaves reproducibly despite not being a true image layer
Manual terminal installs (the Copilot incident; current MRtrix testing)Writable layer, genuinely one-off — lost on delete
Repo folder, ~/.ssh, ~/.gitconfigBind mounts — never in any layer

This is also why the §11-of-the-Runbook "install manually → script it → delete container → replay" plan is a genuine test: deleting the container wipes the writable layer, so if install_extra.sh reproduces the environment afterward, the tools are provably captured in the recipe and not accidentally surviving in transient state.

Nested Docker — how this project uses two levelsnew

The general layer model above describes one Docker environment. This project runs two, nested — Docker-in-Docker — because the deployable artifact (the server/ image) is built from inside the devcontainer. Each level is a complete instance of everything in §11, one level apart.

Outer level — the devcontainer

Image from .devcontainer/Dockerfile (Debian 11, pinned DinD 24.0.0, dev tooling). Its container is what you work inside — own writable layer, own postCreateCommand/install_extra.sh (BART, soon MRtrix). Managed by the host VM's Docker daemon.

Job: give you a working development environment.

Inner level — the server image

A second, separate server/Dockerfile — different FROM, different purpose. Contains Python, inference libs, MRtrix if runtime-required (§10), weights, and the module code. Built by the nested Docker daemon running inside the outer container.

Job: be the actual deployable artifact for the scanner — no dev tooling, no git, nothing from the outer world.

Two independent daemons — which one you're talking to matters: the outer daemon (the host VM's Docker) created your devcontainer. The inner daemon (started by postStartCommand via nohup dockerd &, only running inside the devcontainer) builds and manages the server/Dockerfile image. This is why the devcontainer needs --privileged and explicitly starts its own dockerd — a container doesn't get Docker-inside-it for free.

Everything from §11 still holds at each level independently: server/Dockerfile's RUN/COPY lines produce their own cached image layers; the built image gets its own writable layer when eventually run on the scanner. The nesting adds exactly one new thing to keep track of — the two-daemon distinction above — and nothing else changes about the layer mechanics.

Practical consequence for the MRtrix runtime question (§10): testing whether MRtrix needs to be in server/Dockerfile too happens via this inner daemon, from inside the already-running devcontainer — point docker build at server/Dockerfile instead of .devcontainer/Dockerfile. No need to leave the devcontainer or open a second window; it's the same nested-build path docker_build.sh/create_archive.sh already use.

Module architecture — the logic the frameworks imposenew

The four tiers above answer where things live. This section answers why the code is shaped the way it is — the framework logic that the Setup Runbook's §15–§17 then turn into concrete commands. Two frameworks meet here: the OpenRecon server (a plugin host) and Supervised_RNS_dMRI (a BIDS-style model-fitting tool). The design work is fitting the second cleanly into the first.

The server is a dispatcher; each module is a strategy

The image packages functionality; it does not itself implement it. server.py is a dispatcher: it accepts one TCP connection, selects a single module by OR_MODULE, and hands off to that module's one interface method — process(connection, ui_data, mrd_header). Every module is an interchangeable implementation of that same interface, sharing a common src/ library. In C++ terms this is the strategy pattern: one fixed interface, swappable concrete strategies selected at runtime, with the dispatcher never needing to know which one it holds.

Fig. 3 — plugin dispatch: one interface, interchangeable module implementations

Consequence for adding a module: a new .py file under server/modules/ is swept into the image automatically by the existing COPY line — no Dockerfile edit is needed to add the file itself, only to register its name in server.py's choices= list and (separately) to add any new runtime library it pulls in. This is the same "recipe vs. identity" discipline as the tiers above, one level down: code travels for free with the build context; new dependencies must be declared.

Why i2i_invertcontrast, not r2ci_bart, is the scaffold. The diffusion-microstructure tool is image-to-image: the scanner has already reconstructed the DWI volumes, and the module infers parameter maps from those images. So the module receives a stream of ismrmrd.Image messages, not raw ismrmrd.Acquisition k-space — making the image-to-image module the correct template to copy from. (Command-level integration steps: Runbook §15.)

The core mismatch: streamed volumes vs. one 4D file on disk

MRD sends one image message per diffusion volume/slice; Supervised_RNS_dMRI expects a single 4D NIfTI plus .bval/.bvec sidecars on disk. The module therefore has to buffer the incoming series, stack it once complete, and write it out (via nibabel, already in the dependency chain) before it can call the fitting logic — then convert the returned NIfTI parameter maps back into ismrmrd.Image messages. The bridge, not the science, is where most of the framework-integration effort goes.

Where b-values/gradient directions come from is a real design decision, not a detail. MRD has no dedicated b-value field; on a real scanner this metadata would ride in through vendor/private tags (userParameterLong/Double/String or per-image Meta attributes). Since the hackathon target is a local .h5 test client rather than a live scanner, solving true custom-field extraction is out of scope — but whichever shortcut is chosen (bundling sidecar files, reusing the ui_data JSON channel, or computing mask/noise on the fly with MRtrix) has a scanner-deployability trade-off worth stating explicitly in the module's own Readme. (Options and trade-offs: Runbook §15.3.)

Offline training vs. runtime inference — a hard architectural split

This is the single most important logical boundary in the project, and it maps directly onto the tier model. server.py serves one connection and exits — it is not a warm, long-running service. Re-training a Random Forest/MLP per connection would be both wasteful and architecturally wrong. So training is done offline, once, as preparation, entirely outside the module's process():

Training side — never ships

Runs against a cohort of real scans to characterise realistic noise statistics, with DiffSimGen simulating clean signal/parameter pairs at the target protocol. Produces exactly two frozen pickles: the trained model and a small modelinfo metadata dict. The cohort data, the simulation code path, and the training scripts stay behind — none of it enters the shipped image.

Inference side — what tier 4 contains

Only the two frozen .pkl files plus the inference-side logic (apply_RF_python/apply_MLP_python) and the real-data preprocessing the single incoming dataset needs. These pickles are baked into server/Dockerfile simply by placing them inside the module folder before docker build — the existing COPY sweeps them in, no Dockerfile edit for the model file itself.

Version-matching gotcha, same class as the Python-3.11 bump. Pickles are sensitive to the exact scikit-learn/numpy versions that created them. Train in the same validated devcontainer environment, and make sure server/Dockerfile ends up with matching versions at inference time, or unpickling can fail or misbehave subtly. This decoupling is also the natural place to split students into a training subgroup and an integration subgroup (Runbook §15.5).

MRtrix's role — and why it lands in tier 4

MRtrix3 supplies the brain mask (dwi2mask) and the MP-PCA noise estimate (dwidenoise) that Supervised_RNS_dMRI expects as inputs — the tool loads an already-computed noise map rather than deriving it. Because those steps run inside the module's process() on the incoming dataset, MRtrix is a genuine runtime dependency (tier 4), not merely a dev-time convenience — which resolves the open question in §10 in favour of baking it into server/Dockerfile, pending final confirmation with the tool developers.

How this closes back onto the tiers: training lives entirely at tiers 2/3 (dev environment), the frozen pickles and MRtrix cross into tier 4 (the shipped image), and nothing about the training cohort or dev tooling follows them. The plugin dispatch, the streamed-to-file bridge, and the offline/runtime split are the three pieces of framework logic; the Setup Runbook (§15–§17) is where each becomes an actual command sequence.

§5 and §7 established the principle — dev tooling lives at tiers 2/3, runtime dependencies at tier 4, and the two Dockerfiles must stay strictly separate. This section is that principle made concrete: an actual field-by-field comparison of .devcontainer/Dockerfile against server/Dockerfile. It exists because a step that belongs in one file usually does not belong in the other, and the reasons are structural, not arbitrary — which only becomes obvious once the two are laid side by side.

Why matplotlib==3.9.2 and not 3.10.8, in both files: pinned pre-emptively, because Supervised_RNS_dMRI's own dependencies force <=3.9.2 the moment it's installed. Since matplotlib is touched by neither the reconstruction math nor the NRMSE test suite (only by client/display.py's visualization), there's no reason to let pip silently downgrade it later rather than simply starting there. Same class of decision as the Python 3.12 → 3.11 bump (§3b): resolve the version conflict once, at the recipe level, instead of discovering it mid-install.

Two questions to ask before adding any new dependency

  1. Does this need to survive on the scanner, or is it just for development/testing? Development-only tools (Copilot, git-cola, manual experimentation) belong in .devcontainer/Dockerfile / install_extra.sh — or in no committed file at all, if it's purely a manual exercise (§10). Anything the reconstruction module itself calls at runtime — MRtrix3's command-line tools, Supervised_RNS_dMRI's Python dependencies — must be baked into server/Dockerfile, because that is the only file ever built into the deployed image. There is no postCreateCommand equivalent for it: once packaged and installed on the scanner, there is no manual step left to run.
  2. Given server/Dockerfile has no non-root user, what changes? Every RUN there already executes as root — no sudo prefix. Environment variables should be set with ENV (immediately visible to every process, including docker exec), not via the /etc/profile.d + ~/.bashrc layering that install_extra.sh needs only to work around postCreateCommand's "new shells only" limitation (§3b). New tools should land at a top-level path like /mrtrix3, following the existing /bart convention, rather than nested under a user-owned directory like /container.

Where does a module actually get tested? Three stages, not one

This is the single most important thing to internalise before writing any module code, because "client/server" in OpenRecon sounds like it requires two machines or two containers. It doesn't. Both are simply two local processes that talk over a TCP socket (port 9002), and both can run side by side inside the same devcontainer. server/Dockerfile only enters the picture once the actual deployment image needs validating — a separate, later, slower stage.

StageHow it runsWhat it provesWhen to use it
1 · Host-mode ./run_server.sh <module> + ./run_client.sh in two terminals, or the VS Code "Server + client" debug profile (F5) The module's own logic — process(), the science, the MRD message handling Almost always. Fastest loop: no Docker build, breakpoints work, both processes run natively against the devcontainer's own Python/BART/MRtrix3.
2 · Docker-mode ./run_docker.sh <module> (Docker-in-Docker, via the inner daemon of §12) That the packaged image behaves identically — i.e. that everything the module needs is actually baked into server/Dockerfile, not merely present in the devcontainer Occasionally, once Stage 1 is clean. This is where a missing server/Dockerfile dependency first surfaces as a real failure. No extra machine needed.
3 · Packaging ./create_archive.sh Produces the actual scanner-installable .zip Once, near the end — not something re-run per code change.
The exact failure mode this three-stage split exists to prevent: writing and testing a module entirely in Stage 1, where everything works because the devcontainer has every tool manually installed — and only then discovering, at Stage 2 or (far worse) on the actual scanner, that server/Dockerfile never received the same dependency, because that step was quietly skipped. Stage 2 exists for precisely this gap. It is the direct operational consequence of §11's layer model: what lives in a devcontainer's writable layer is invisible to the deployed image, and no amount of Stage-1 success will reveal that.

Step-by-step commands for Stages 1–3 live in the Setup Runbook (§13, §16, §17); this section is the why behind them. A fuller line-by-line reference is kept alongside the code as docs/Dockerfile-Comparison.md.
