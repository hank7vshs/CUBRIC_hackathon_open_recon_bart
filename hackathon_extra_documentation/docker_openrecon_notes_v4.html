01

VM vs. container

These virtualize different things, which is the whole source of confusion at the start.

Virtual machine

Emulates a whole computer: its own kernel, OS, virtual hardware. Heavy, strongly isolated. Like a separate house built on your land — own foundation, own plumbing.

Container

Shares the host's kernel; isolates only the userspace — filesystem, libraries, process view. Light, starts in seconds. Like an apartment in a building — own door, shared foundation.

Image is to container as class is to instance. The image is the frozen blueprint; the container is a running instance of it.

The hosting VM sits one level up again: it is the Ubuntu machine on which the Docker engine runs. Containers borrow that VM's kernel rather than booting one of their own — which is exactly why a container starts in seconds and why anything installed by hand inside it lives only in that VM's Docker state until it is captured in a Dockerfile. §04 makes this VM → engine → devcontainer → nested container nesting concrete.

02

Image → container lifecycle

An image is a stack of read-only layers — a frozen filesystem plus metadata. It gets built two ways, and turned into a container by one command:

Fig. 1 — pull or build both yield an image; run adds a writable layer to make a container

Changes made by hand inside a running container land only in that writable layer — real, but ephemeral and non-reproducible. docker commit can freeze that layer into a new image, but the result is a black box with no recoverable build recipe.

03

Dockerfile as source of truth

The proper way to "modify" an image is to change the recipe and rebuild, not to hand-edit a running container.

FROM <base-image>              # inheritance
RUN pip install ...             # dependencies — new layer
COPY server/ /app/               # your code — new layer
CMD ["python3", "server.py"]     # what runs on container start
Workflow that reconciles both worlds: experiment freely inside a live container to find the commands that work, then transcribe those commands into the Dockerfile and rebuild from scratch to confirm reproducibility. The container is the scratchpad; the Dockerfile is what gets committed.

In this project the FROM line is concretely python:3.12-slim-bullseye — i.e. Debian 11, not Ubuntu — and the same base is shared by the devcontainer and the shipped server/ image so both run on the same interpreter (bumped to Python 3.11 for the diffusion dependency chain; see the Framework Description doc, §3b). The Docker-in-Docker version is pinned to 24.0.0 deliberately, to match the VA60 scanner's MaRS GPU baseline.

04

The devcontainer layer: logic and integration with Dockerre-added in v3

"Devcontainer" is not a separate technology from Docker — it is a layer built on top of it. Docker alone only knows about images, containers, and layers; it has no concept of "editor," "extension," or "developer workflow." The devcontainer spec exists specifically to fill that gap.

Docker, on its own

docker build, docker run, images, containers, mounts — but only if the flags are typed by hand every time. General-purpose, no notion of a developer sitting inside the container.

Devcontainer, on top

A declarative file, devcontainer.json, read by a client tool that translates it into ordinary Docker commands: which Dockerfile to build, which folders to mount, which extensions to install, which ports to forward, and lifecycle commands to run.

Fig. 2 — devcontainer.json as an orchestration layer that drives the Docker engine

  • Open specification, not a VS Code trick. The devcontainer format lives at containers.dev and is implemented by several tools — VS Code's Dev Containers extension, GitHub Codespaces, JetBrains Gateway, and a standalone devcontainer CLI usable without any editor.
  • At the bottom, it is still just Docker. Everything devcontainer.json declares eventually becomes an ordinary docker build plus an ordinary docker run with flags (-v for mounts, -p for ports) that could be typed by hand — the spec exists so that configuration is written once, checked into git, and reproduced identically by anyone who opens the folder.
  • Builder / facade relationship. Docker is the low-level API — flexible but verbose. devcontainer.json is a declarative recipe that a facade (the extension or CLI) reads and translates into the correct sequence of low-level Docker calls, so nobody has to assemble those calls by hand.
In this repo, concretely: .devcontainer/Dockerfile is pure Docker — same recipe concept as everywhere else in this document. .devcontainer/devcontainer.json is the extra layer that tells VS Code how to use that Dockerfile: build it this way, mount the repo here, forward this port, install these extensions, run this command after creation. Delete devcontainer.json entirely and the Dockerfile still builds and runs fine with plain docker build/docker run — only the automatic editor wiring that makes "Reopen in Container" a one-click action would be lost.

From ephemeral trial to permanent recipev4

The devcontainer's whole purpose is to be an ephemeral testbed: install something by hand, see if it works, keep going. Nothing about that testing step is automatically remembered by any image. Getting a validated install to stick — so every student's build has it too — requires one explicit, manual action: copy the command into the Dockerfile and rebuild.

Fig. 3 — an ephemeral install only becomes reproducible once it's transcribed into the Dockerfile and rebuilt

docker diff <container> lists everything added, changed, or deleted in the writable layer versus the image it started from — useful for double-checking nothing was forgotten before transcribing, though it won't write the Dockerfile lines for you. Reverse-engineering tools that attempt this from a committed image exist but are heuristic and lossy (see §3's note on docker commit) — not something to build a reproducible pipeline on.

Rebuild ≠ patch

A rebuild does not take the old image and bolt new layers onto it. Docker builds fresh from the top of the current, full text of the Dockerfile every time. What actually changes between builds is the recipe, not the previous image.

Where new lines go

New RUN/COPY lines are usually added lower in the file — not because correctness requires it, but because of the build cache: everything after a changed line gets rebuilt, so rarely-changing steps (base image, OS packages) belong near the top, frequently-edited ones (your code) near the bottom. CMD/ENTRYPOINT stay last regardless, since that's what actually runs on container start.

Two different things get added to the recipe, and only one goes through the "trial → transcribe" cycle: dependencies (MRtrix, PyTorch, a pip package) are installed, so they need to be validated live first, then written as a RUN line. Your own module code, trained weights, and auxiliary data are not installed at all — they're copied. The Dockerfile's own COPY instruction is what pulls in that file tree directly from the repo; there's no live-testing phase for that step, since it's not a package install, just your own files being placed into the image at build time. COPY server/ /app/ already covers any new file added under that folder, with no Dockerfile edit needed for the file itself.
05

The nested stack in this repo

Applied to the actual environment: an Ubuntu VM running a Docker engine, inside which VS Code's devcontainer runs, inside which the Open Recon container can itself be built and run (Docker‑in‑Docker).

Fig. 4 — VM → Docker engine → devcontainer → nested Open Recon container

VS Code itself splits at this boundary: the window renders on the host, but the terminal, extensions, and debugger run inside the devcontainer. That is why installing a tool like Claude Code on the host does not make it visible inside the container.

06

Open Recon's plugin architecture

The image packages functionality; it does not itself implement it. server.py is a dispatcher; each module is an interchangeable strategy implementing one interface — process(connection, ui_data, mrd_header).

Fig. 5 — plugin dispatch: one interface, interchangeable module implementations

For the hackathon target: the diffusion-microstructure tool is image-to-image (scanner has already reconstructed; the tool infers parameter maps from those images) — so i2i_invertcontrast is the right scaffolding, not r2ci_bart.

  • Copy i2i_invertcontrast as a starting module; replace its trivial contrast-inversion line with the RNS-trained inference call.
  • Register the new module name in server.py's choices= list.
  • A new module's .py file is picked up automatically on the next image build — COPY server/ /app/ copies the whole directory, so no Dockerfile edit is needed to "add" the file itself.
  • A Dockerfile edit is needed only when a new library is required (e.g. PyTorch for the inference model) — add it to both Dockerfiles (see §8).
07

Two sharing modes

The confusion dissolves once you separate: share to rebuild vs. share to run. Different goal, different artifact.

GoalWhat you shareWho uses it, how
Collaborate / reproduce / reviewThe git repo — Dockerfile + src/ + modules/ + data, versioned togetherRecipient clones and runs docker build themselves
Deploy / hand someone a working toolThe built image — docker save to a .tar, or a registry pushRecipient runs docker load + run; no source needed

A Dockerfile alone, without the files it COPYs, fails to build for anyone else — the shareable unit is the whole build context, not the single file. For the hackathon specifically: the repo is what goes to the tool developers and Open Recon tutors; the packaged image (via create_archive.sh) is what goes to the Free.Max scanner.

08

Two Dockerfiles, two audiences

.devcontainer/Dockerfile

Your workshop. Editors, git, AI assistants (Claude Code, Copilot), CUDA, BART for dev. Never ships to the scanner.

server/Dockerfile

Exclusively runtime dependencies process() needs on the scanner. No dev tooling belongs here — it would bloat the deployed artifact.

Rule of thumb: tool goes in the recipe, identity gets mounted at runtime. Git binary → baked into the devcontainer image. Git credentials, SSH keys, ~/.claude auth → bind-mounted from the host, never baked into any image.
09

VS Code + Claude Code in the devcontainer

Opening "Reopen in Container" splits VS Code: the UI stays on the host, the backend (terminal, extensions, debugger) relocates inside the container. Anything typed in that terminal runs with the container's tools — which is why a host-installed CLI is invisible there.

To have Claude Code available inside the devcontainer, reproducibly:

// .devcontainer/devcontainer.json
"features": {
  "ghcr.io/devcontainers/features/node:1": {},
  "ghcr.io/anthropics/devcontainer-features/claude-code:1.0": {}
}

Node.js is a prerequisite for the Claude Code feature. Bind-mount ~/.claude from the host to persist auth/settings across rebuilds, the same pattern used for git credentials.
